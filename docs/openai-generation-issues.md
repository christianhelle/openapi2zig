# OpenAI SDK generation status

Validation timestamp: 2026-04-28T23:01:58.137+02:00

This document tracks OpenAI-style real-world generation concerns after PR #46, "Fix real-world OpenAPI code generation". It replaces the older issue list that described missing raw responses, streaming helpers, `extra_body`, and resource wrappers before those features landed.

## Current status

The generator now produces usable single-file clients for large OpenAPI documents with these behaviors:

- `Client` includes allocator, `std.Io`, owned `std.http.Client`, API key, base URL, optional organization/project headers, and borrowed `default_headers`.
- Parsed endpoint calls return `Owned(T)` so response JSON storage stays alive until callers call `deinit()`.
- Endpoint-specific raw helpers such as `createResponseRaw(...)` return `RawResponse` with status and body.
- Endpoint-specific result helpers such as `createResponseResult(...)` return `ApiResult(T)` and preserve non-2xx bodies and parse-error bodies.
- Generated response parsing uses `.ignore_unknown_fields = true` by default.
- Generic helpers include `requestRaw`, `getRaw`, `postJsonRaw`, `getJsonResult`, and `postJsonResult`.
- Bounded SSE helpers support raw and typed callbacks: `parseSseBytes`, `parseSseReader`, `parseSseBytesTyped`, and `parseSseReaderTyped`.
- Streaming request helpers set `stream: true` and use `Accept: text/event-stream` where generated for streaming operations.
- Extensible request structs can include `extra_body`; generated JSON serialization flattens `extra_body.object` into the root request body.
- Provider extension fields such as `reasoning_details` are supported where the generator recognizes the request/response shape.
- Resource wrappers are generated by default from path metadata, with CLI modes `none`, `tags`, `paths`, and `hybrid`.
- `std.json.Value` is still used deliberately for ambiguous, open-ended, or dynamic JSON schema shapes.

## Remaining focus areas

### Multipart and binary ergonomics

Some real-world endpoints use `multipart/form-data`, file uploads, file downloads, or non-JSON response bodies. JSON request/response generation is stronger than multipart and binary ergonomics today.

Acceptance:

- Inspect operation request content type before choosing JSON serialization.
- Generate multipart runtime support for `multipart/form-data` request bodies.
- Return raw or binary-friendly responses for file/audio endpoints.
- Avoid forcing `application/json` for multipart-only operations.

### `std.json.Value` reduction

The current policy intentionally keeps `std.json.Value` for ambiguous and open-ended schemas, but important provider schemas can still benefit from more concrete generated types.

Acceptance:

- Keep `std.json.Value` for genuinely dynamic JSON schema fields and extension maps.
- Prefer generated structs, arrays, and safe unions when the schema gives enough shape.
- Continue improving discriminated and structural union generation without removing raw fallbacks.

### Resource wrapper naming

Resource wrappers now exist, but names are generated from path/tag metadata and may not always match hand-designed SDK shapes.

Acceptance:

- Keep flat functions as the stable compatibility surface.
- Use `--resource-wrappers none` for flat-only output.
- Improve wrapper method names and nesting when path/tag metadata is clear.
- Avoid breaking existing generated flat endpoint names.

## Current caller patterns

Use convenience methods when only successful parsed responses matter:

```zig
var client = api.Client.init(allocator, io, api_key);
defer client.deinit();
client.withBaseUrl("https://api.openai.com/v1");

var response = try api.createResponse(&client, request_body);
defer response.deinit();
```

Use result helpers when callers need response status/body for API errors or parse failures:

```zig
var result = try api.createResponseResult(&client, request_body);
defer result.deinit();

switch (result) {
    .ok => |*ok| try useResponse(ok.value()),
    .api_error => |raw| std.debug.print("HTTP {}: {s}\\n", .{ raw.status, raw.body }),
    .parse_error => |parse| std.debug.print("parse {s}: {s}\\n", .{ parse.error_name, parse.raw.body }),
}
```

Use raw helpers when the endpoint response is not JSON or the caller wants full control:

```zig
var raw = try api.createResponseRaw(&client, request_body);
defer raw.deinit();
std.debug.print("HTTP {}\\n{s}\\n", .{ raw.status, raw.body });
```
